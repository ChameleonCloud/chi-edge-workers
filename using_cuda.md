# CUDA and GPU Support on CHI@Edge

## Overview

CHI@Edge runs k3s inside Balena containers on Jetson devices. The NVIDIA
container runtime mounts GPU devices and libraries from the k3s host container
into user pods. Users request GPUs via standard Kubernetes resource limits.

There are two JetPack generations in the fleet:

| Platform | JetPack | L4T | CUDA | Compute Capability |
|---|---|---|---|---|
| Jetson Nano | 4 (r32.7) | 32.7.3 | 10.2 | 5.3 |
| Xavier NX | 4 (r32.7) | 32.7.3 | 10.2 | 7.2 |
| Orin Nano | 6 (r36.4) | 36.4 | 12.6 | 8.7 |

## Important Notes and Footguns

- **CUDA version mismatch is the #1 failure mode.** Binaries compiled against
  CUDA 12 will not run on JetPack 4 devices (CUDA 10.2) and vice versa. The
  error is `libcublas.so.12: cannot open shared object file`.

- **Pods must request `nvidia.com/gpu: "1"` to get GPU access.** Even though
  the nvidia runtime is the default, it only mounts devices and libraries into
  pods that explicitly request a GPU resource. Without the resource limit, pods
  see no GPU devices.

- **The L4T userspace version must match the BalenaOS kernel.** The k3s
  container packages the entire NVIDIA userspace (drivers, CUDA libs, container
  runtime). These must match the L4T version of the kernel running on the host.
  A mismatch causes silent failures or `NvRmMemInitNvmap` errors. Check with
  `uname -r` on the Balena host — the kernel string contains the L4T version
  (e.g. `4.9.299-l4t-r32.7.3`, `5.15.148-l4t-r36.4`).

- **`nvidia-smi` only exists on JetPack 6.** On JetPack 4, the equivalent is
  `tegrastats`. Don't use `nvidia-smi` as a health check on Nano/Xavier.

- **The container toolkit must come from `nvidia.github.io/libnvidia-container`,
  not from the Jetson L4T repo.** The L4T repo (`repo.download.nvidia.com/jetson/common`)
  has an older toolkit (1.7.0) that lacks CSV/CDI support. The official toolkit
  repo has 1.19.0+ which works on both JetPack 4 and 6. The L4T repo is only
  for BSP packages (`nvidia-l4t-*`).

- **CDI specs must be generated at startup.** The nvidia container runtime uses
  CDI (Container Device Interface) specs to know what to mount into pods. These
  are generated by `nvidia-ctk cdi generate` and written to `/var/run/cdi/`.
  This path is ephemeral — the spec must be regenerated every time the k3s
  container starts. This is done in `start.sh`.

- **`nvidia-container-cli info` may fail even when the runtime works.** On
  JetPack 6 (Orin), the CLI's GPU detection uses `libnvrm_gpu.so` which fails
  with `NvRmMemInitNvmap Permission denied` inside nested containers. This does
  not affect pod GPU access — the CDI spec bypasses the CLI's detection path.

- **`runc` must be installed separately.** k3s bundles its own runc in a
  versioned data directory that isn't on PATH. The nvidia container runtime
  needs to find runc to wrap it. Install `runc` from apt in the k3s Dockerfile.

- **The `nvidia-container-csv-cuda` package depends on the full CUDA toolkit
  (~3GB).** On JetPack 4, use `cuda-libraries-10-2` (425MB, runtime libs only)
  and write the `cuda.csv` file manually. It's one line: `dir, /usr/local/cuda-10.2`.

## 1. How GPU Support is Enabled in the Base Images

### Package sources

Two apt repos are configured in the Jetson base image Dockerfiles:

1. **L4T BSP repo** (`repo.download.nvidia.com/jetson/{SOC}` and `.../common`):
   Provides `nvidia-l4t-cuda`, `nvidia-l4t-firmware`, `nvidia-l4t-init` — the
   Tegra driver stack and CUDA driver (`libcuda.so`).

2. **Container toolkit repo** (`nvidia.github.io/libnvidia-container/stable/deb/`):
   Provides `nvidia-container-toolkit` (1.19.0+), which includes
   `nvidia-container-runtime`, `nvidia-ctk`, `libnvidia-container`, and the
   runtime hook.

### JetPack 4 (Nano, Xavier NX)

- Base: `ubuntu:18.04`
- BSP packages: `nvidia-l4t-cuda`, `nvidia-l4t-firmware`, `nvidia-l4t-init`
- CUDA runtime: `cuda-libraries-10-2` (installed to `/usr/local/cuda-10.2/`)
- CSV file: hand-written `cuda.csv` pointing at `/usr/local/cuda-10.2`
- The `l4t.csv` file comes from the L4T BSP packages and lists Tegra devices
  and libraries

### JetPack 6 (Orin Nano)

- Base: `ubuntu:22.04`
- BSP packages: `nvidia-l4t-cuda`, `nvidia-l4t-firmware`, `nvidia-l4t-init`
- Additionally needs `nvidia-l4t-nvml` for `libnvidia-ml.so.1` and `nvidia-smi`
- No CSV files needed — CDI handles everything

### k3s Dockerfiles

The k3s Dockerfiles add `iproute2`, `runc`, and the k3s binary on top of the
base image. Jetson variants pass `--default-runtime=nvidia` to k3s via CMD.

### start.sh

Before launching k3s, `start.sh` generates the CDI spec if `nvidia-ctk` is
available:

```sh
if command -v nvidia-ctk >/dev/null 2>&1; then
  nvidia-ctk cdi generate --output=/var/run/cdi/nvidia.yaml
fi
```

On JetPack 4 this auto-detects CSV mode and reads the CSV files. On JetPack 6
it uses direct device enumeration.

## 2. How the NVIDIA Runtime and Device Plugin Work

### Container runtime chain

```
containerd (inside k3s)
  → nvidia-container-runtime (wraps runc)
    → nvidia-container-runtime-hook (prestart hook)
      → reads CDI spec from /var/run/cdi/nvidia.yaml
      → bind-mounts GPU devices and libraries into the pod
        → runc (creates the actual container)
```

### containerd configuration

k3s generates `/var/lib/rancher/k3s/agent/etc/containerd/config.toml` with:

```toml
[plugins."io.containerd.grpc.v1.cri".containerd]
  default_runtime_name = "nvidia"

[plugins."io.containerd.grpc.v1.cri".containerd.runtimes."nvidia"]
  runtime_type = "io.containerd.runc.v2"
[plugins."io.containerd.grpc.v1.cri".containerd.runtimes."nvidia".options]
  BinaryName = "/usr/bin/nvidia-container-runtime"
```

This is configured by passing `--default-runtime=nvidia` to `k3s agent`.

### CDI spec

The CDI spec (`/var/run/cdi/nvidia.yaml`) is generated by `nvidia-ctk cdi generate`
at startup. It lists:

- GPU device nodes (`/dev/nvhost-*`, `/dev/nvidia*`, `/dev/nvmap`)
- Library paths to bind-mount (`/usr/lib/aarch64-linux-gnu/tegra/libcuda.so.1.1`, etc.)
- Firmware paths (`/lib/firmware/tegra*`)
- Symlinks and ld.so.conf entries

On JetPack 4, `nvidia-ctk` reads CSV files from
`/etc/nvidia-container-runtime/host-files-for-container.d/` to build this spec.
On JetPack 6, it enumerates devices directly (with some CSV fallback via
`--mode=csv`).

### NVIDIA device plugin

The [NVIDIA device plugin](https://github.com/NVIDIA/k8s-device-plugin) runs
as a DaemonSet on GPU nodes. It:

1. Discovers GPUs on the node
2. Advertises `nvidia.com/gpu` as a schedulable resource to the kubelet
3. Manages GPU allocation when pods request `nvidia.com/gpu`

Nodes must be labeled `nvidia-device-plugin/enabled=true` for the DaemonSet
to target them:

```bash
kubectl label node <node-name> nvidia-device-plugin/enabled=true
```

### What gets mounted into pods

When a pod requests `nvidia.com/gpu: "1"`, the runtime mounts:

- Device nodes: `/dev/nvhost-*`, `/dev/nvidia*`, `/dev/nvmap`, etc.
- Libraries: `libcuda.so`, tegra driver libs, EGL libs
- Firmware: `/lib/firmware/tegra*`
- On JetPack 4: CUDA runtime libs from `/usr/local/cuda-10.2/`

The pod does NOT need CUDA libraries in its own image — they are mounted from
the k3s host container.

## 3. Operator: Verifying GPU Support

### Check node labels

```bash
kubectl get nodes --show-labels | grep nvidia-device-plugin
```

### Check GPU resource is advertised

```bash
kubectl describe node <node-name> | grep -A5 "Capacity\|Allocatable"
# Should show: nvidia.com/gpu: 1
```

### Check containerd config (inside k3s container)

```bash
cat /var/lib/rancher/k3s/agent/etc/containerd/config.toml | grep -A3 nvidia
# Should show default_runtime_name = "nvidia" and BinaryName = nvidia-container-runtime
```

### Check CDI spec exists

```bash
cat /var/run/cdi/nvidia.yaml | head -20
```

### Check runc is available

```bash
which runc
# Should return /usr/sbin/runc or similar
```

### Quick GPU test

```bash
kubectl run gpu-test --rm -it --restart=Never --image=ubuntu \
  --overrides='{"spec":{"nodeName":"<node>","containers":[{"name":"t","image":"ubuntu","command":["ls","/dev/nvhost-ctrl-gpu"],"resources":{"limits":{"nvidia.com/gpu":"1"}}}]}}'
```

### On JetPack 6 only

```bash
# Inside k3s container:
nvidia-smi
```

### What to check when GPU doesn't work

1. Is `nvidia-ctk` on PATH? (`which nvidia-ctk`) — if not, the container
   toolkit repo wasn't added or the package version is too old (< 1.19.0).
2. Does the CDI spec exist? (`ls /var/run/cdi/`) — if not, `nvidia-ctk cdi generate`
   failed at startup. Check k3s container logs.
3. Is `runc` on PATH? (`which runc`) — if not, the nvidia runtime can't wrap it.
4. Is the device plugin running? (`kubectl get pods -A | grep nvidia-device-plugin`)
5. Does the node advertise GPU? (`kubectl describe node | grep nvidia.com/gpu`)
6. Check k3s agent startup for errors about nvidia runtime.

## 4. End-User: Using GPU in Pods

### Requesting a GPU

Add a resource limit to your pod spec:

```yaml
resources:
  limits:
    nvidia.com/gpu: "1"
```

Without this, your pod will not have GPU access even on a GPU node.

### Choosing a base image

Your container image does NOT need to include CUDA libraries — they are
mounted by the runtime. However, if you compile CUDA code, you must target
the correct CUDA version for the device.

| Device | CUDA Version | Compile with |
|---|---|---|
| Jetson Nano | 10.2 | `nvcc -arch=sm_53` |
| Xavier NX | 10.2 | `nvcc -arch=sm_72` |
| Orin Nano | 12.6 | `nvcc -arch=sm_87` |

**Critical:** If you compile against CUDA 12 and your pod lands on a CUDA 10.2
node, it will fail with a missing shared library error. Target the lowest CUDA
version your code supports, or build separate images per device type.

For pre-built images, NVIDIA provides:
- JetPack 4 (CUDA 10.2): `nvcr.io/nvidia/l4t-cuda:10.2.460-runtime`
- JetPack 6 (CUDA 12.2): `nvcr.io/nvidia/l4t-cuda:12.2.12-devel`

Or use generic Ubuntu/Debian images — CUDA libs are mounted at runtime.

### Targeting specific devices

Use a nodeSelector to ensure your pod runs on the right hardware:

```yaml
spec:
  nodeSelector:
    kubernetes.io/hostname: iot-orin-nano-01
```

Or use node labels/affinity for broader targeting.

### Environment variables

The runtime respects `NVIDIA_VISIBLE_DEVICES`. Set it to `all` to expose all
GPUs (default behavior when `nvidia.com/gpu` is requested):

```yaml
env:
  - name: NVIDIA_VISIBLE_DEVICES
    value: "all"
```

## 5. End-User: Verifying GPU Access

### Check devices are visible

```bash
ls /dev/nvhost-ctrl-gpu
```

If this file doesn't exist, the GPU was not mounted. Check that your pod
spec includes `nvidia.com/gpu: "1"` in resource limits.

### Check CUDA works

```bash
# Inside your pod:
python3 -c "import ctypes; ctypes.CDLL('libcuda.so.1'); print('CUDA OK')"
```

Or with PyTorch:

```python
import torch
print(torch.cuda.is_available())
print(torch.cuda.get_device_name(0))
```

### Common errors

| Error | Cause | Fix |
|---|---|---|
| `libcuda.so.1: cannot open shared object file` | GPU not requested in pod spec | Add `nvidia.com/gpu: "1"` to resource limits |
| `libcublas.so.12: cannot open shared object file` | CUDA version mismatch | Recompile against CUDA 10.2 for JP4 devices |
| `no CUDA-capable device is detected` | Runtime mounted devices but driver mismatch | Check L4T version matches between image and host |
| `/dev/nvhost-ctrl-gpu: No such file or directory` | GPU not mounted | Verify nvidia-device-plugin is running and node has `nvidia.com/gpu` resource |
